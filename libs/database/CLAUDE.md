# Database Library

Prisma schema: `prisma/schema.prisma`. `DatabaseModule` is `@Global()` — only import in root `AppModule`.

## Key Files

| File                                  | Purpose                                                                                |
| ------------------------------------- | -------------------------------------------------------------------------------------- |
| `database-connection.ts`              | PrismaClient subclass with `pg.Pool` adapter, lifecycle hooks, pool config from env    |
| `database.service.ts`                 | DI token — PrismaClient extended with soft-delete, augmented types for `deletedBy`     |
| `hard-delete.service.ts`              | DI token — raw PrismaClient WITHOUT soft-delete extension (for permanent deletes)      |
| `database.module.ts`                  | Wires `DatabaseConnection` → `withSoftDelete()` → `DatabaseService` via factory        |
| `extensions/soft-delete.extension.ts` | Prisma `$extends` that auto-injects `isDeleted: false` and converts deletes to updates |

## Prisma Conventions

- **Model names**: PascalCase, with `@@map("snake_case_table_name")`
- **Field names**: camelCase, with `@map("snake_case")` where they differ
- **ID fields**: `@default(autoincrement())`
- **Timestamps**: every model must have `createdAt` + `updatedAt` with `@map`
- **Foreign keys**: must have `@index` for performance
- **Relations**: explicit `@relation` with `fields` and `references`
- **Enums**: PascalCase with `@@map("snake_case")`

### Exceptions

- **AuditLog** deliberately omits `updatedAt` — audit logs are immutable append-only records

## Soft-Delete Extension

Models with `isDeleted`, `deletedAt`, `deletedBy` fields are auto-detected — no hardcoded list.

**Intercepted operations (13 total):**

- `findFirst`, `findFirstOrThrow`, `findMany`, `count`, `aggregate`, `groupBy`, `updateMany` → auto-inject `isDeleted: false` directly in `where`
- `findUnique`, `findUniqueOrThrow`, `update`, `upsert` → inject `isDeleted: false` via `AND` clause (unique fields constraint requires this approach)
- `delete` → converted to `update` setting `isDeleted: true`, `deletedAt: new Date()`, `deletedBy` — requires `deletedBy` arg
- `deleteMany` → converted to `updateMany` setting `isDeleted: true`, `deletedAt: new Date()`, `deletedBy` — requires `deletedBy` arg
- **Escape hatch**: pass `{ isDeleted: true }` explicitly to query deleted records
- **Not intercepted**: `create`, `createMany` — these don't need filtering

**Usage:**

```typescript
// Soft delete (requires deletedBy)
await this.db.user.delete({ where: { id }, deletedBy: userId } as never);
await this.db.caseAgency.deleteMany({ where: { caseId }, deletedBy: userId } as never);

// Inside $transaction — use `as never` because tx is raw TransactionClient without augmented types
await tx.caseAgency.deleteMany({ where: { caseId }, deletedBy: userId } as never);

// Query deleted records (escape hatch)
await this.db.user.findMany({ where: { isDeleted: true } });
```

**Do NOT** manually add `isDeleted: false` to queries — the extension handles it automatically.

## HardDeleteService

Raw PrismaClient without the soft-delete extension. Use for permanent deletes:

```typescript
constructor(private readonly hardDelete: HardDeleteService) {}
await this.hardDelete.user.delete({ where: { id } }); // true DELETE, no deletedBy needed
```

## DatabaseService Type Augmentation

`DatabaseService` augments `delete`/`deleteMany` on soft-delete models to require `deletedBy: number`. This is type-safe via Prisma's TypeMap — adding `isDeleted`/`deletedAt`/`deletedBy` to a new model auto-includes it.

## Connection & Pool

- Uses `@prisma/adapter-pg` with a manual `pg.Pool` (not Prisma's built-in connection)
- Pool config (`max`, `idleTimeoutMillis`, `connectionTimeoutMillis`) from env via `ConfigService`
- Transaction config (`maxWait`, `timeout`) from env
- Pool cleanup handled by `DatabaseConnection.onModuleDestroy()` — use `app.close()` in tests, not `db.$disconnect()`

## Seeders

- `SeedRunner` runs all seeders on app startup, skipping unchanged seeds (hash-based)
- `RbacSeeder` syncs roles and permissions from `seed-data/` — uses `HardDeleteService` for upserts
- Seed data: `permissions.seed.ts`, `roles.seed.ts`

## Workflow

1. Edit `prisma/schema.prisma`
2. `pnpm prisma:generate` — regenerate client
3. `pnpm prisma:migrate:dev -- --name snake_case_description` — create migration
4. Verify `pnpm exec nx typecheck database`

**NEVER manually write or edit migration SQL files** — they are generated by `prisma migrate dev`. Hand-editing causes drift between the schema and migration history, breaking future migrations.
